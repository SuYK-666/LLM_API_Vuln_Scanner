# LLM API 越权漏洞分析报告

- 测试接口数: 8
- 发现风险数: 8
- 总体风险: High
- 高/中/低: High=3, Medium=2, Low=1

## 风险明细
### 风险项 1

#### 风险对象
- 接口: GET http://127.0.0.1:8000/api/v1/order/detail?order_id=50001

#### 触发条件
- 使用了名为 order-id-increment 的攻击载荷，测试理由：通过顺序ID枚举实现横向权限提升。；具体覆写参数：query={'order_id': '50002'}
- 触发载荷: order-id-increment

#### 风险证据（重点）
- 状态码对比: baseline=200 / attack=200
- 内容相似度: 0.94
- 敏感命中: {'phone': 1}
- JSON 结构重合度:
  - shared_keys=7
  - baseline_keys=7
  - attack_keys=7
  - overlap_ratio=1.00
- 证据摘要: 检测到越权请求成功返回200，并从中正则匹配到了未授权的手机号记录；敏感字段类型：phone；命中统计：phone=1。
- 判定理由:
  - 基线请求和攻击请求均返回200状态码。
  - 检测到高响应体相似度（0.94）。
  - 基线请求和攻击请求的JSON键结构高度一致（共享键=7/7，重叠度=1.00）。
  - 敏感数据模式匹配成功：{'phone': 1}。
- 基线响应预览:
  - {"code":0,"data":{"amount":199.0,"order_id":"50001","phone":"13800138001","uid":"1001"},"msg":"ok"} 
- 攻击响应预览:
  - {"code":0,"data":{"amount":499.0,"order_id":"50002","phone":"13900139002","uid":"1002"},"msg":"ok"} 

#### 影响分析
- 攻击者可通过枚举参数，未授权获取系统内其他用户的 phone 等高价值敏感隐私数据，可能导致大规模数据泄露。

#### 展示结论
- 综合判定风险等级为：高。漏洞利用成功，存在明显的越权行为。

### 防护与缓解建议 (Mitigation & Prevention)
- 摘要: 通过顺序订单ID枚举实现的横向权限提升允许未授权访问其他用户的订单详情。该API端点缺乏服务器端所有权验证，返回了属于不同用户的订单的敏感数据（电话号码）。
- 即时修复:
  - 添加服务器端所有权检查：在返回数据前验证认证用户ID是否与订单的用户ID匹配
  - 实施默认拒绝的授权策略：除非明确的所有权/RBAC验证通过，否则拒绝所有请求
  - 将数字订单ID替换为加密随机UUID以减少可枚举性
  - 为每个/order/detail请求添加验证用户所有权的授权中间件
- 工程加固:
  - 为所有敏感端点实施带有RBAC/ABAC策略的集中式授权服务/中间件
  - 设计API通过路径参数接受资源标识符（例如/orders/{order_id}）并内置所有权验证
  - 使用带有嵌入用户ID的签名令牌（JWT）并强制执行服务器端会话验证
  - 为常见模式创建可复用的授权工具（所有权检查、角色检查、权限检查）
  - 实施基于资源的访问控制，其中每个资源存储所有者ID和访问策略
- 监控与检测:
  - 为所有订单访问尝试添加审计日志记录，包括：用户ID、请求的订单ID、所有权验证结果
  - 实施实时告警机制，针对失败的所有权检查或可疑访问模式（快速ID枚举）
  - 创建安全单元测试，验证当用户尝试访问他人订单时的授权失败情况
  - 监控API日志中的横向权限提升模式：跨不同用户账户的顺序ID访问
  - 实施模拟认证用户尝试访问未授权资源的集成测试
- 建议来源: llm_mitigation_pass

#### 触发条件
- 使用了名为 uid-increment 的攻击载荷，测试理由：水平权限提升：通过数字ID递增访问其他用户资料。；具体覆写参数：query={'uid': '1002'}

  - baseline_keys=9
  - attack_keys=9
  - overlap_ratio=1.00

- 判定理由:
  - 基线请求与攻击请求均返回200状态码。
  - 检测到中等程度的响应体相似度（0.83）。
  - 基线请求与攻击请求的JSON键结构高度一致（共享键=9/9，重叠度=1.00）。

- 基线响应预览:
  - {"code":0,"data":{"address":"Shanghai Pudong District","email":"zhangsan@example.com","id_card":"110101199001011234","name":"Zhang San","phone":"13800138001","uid":"1001"},"msg":"ok"} 
- 攻击响应预览:
  - {"code":0,"data":{"address":"Beijing Haidian District","email":"lisi@example.com","id_card":"110101199202023456","name":"Li Si","phone":"13900139002","uid":"1002"},"msg":"ok"} 

#### 影响分析
- 攻击者可通过枚举参数，未授权获取系统内其他用户的 phone、email、id_card 等高价值敏感隐私数据，可能导致大规模数据泄露。

#### 展示结论
- 综合判定风险等级为：高。漏洞利用成功，存在明显的越权行为。

### 防护与缓解建议 (Mitigation & Prevention)
- 摘要: 通过可预测的数字用户ID参数实现水平权限提升，允许未授权访问其他用户的敏感资料数据（phone、email、id_card）。
- 即时修复:
  - 添加服务器端所有权验证：在返回数据前，验证来自JWT/会话的已认证用户ID是否与请求的uid参数匹配。
  - 将API参数中的顺序数字用户标识符替换为随机UUID。
  - 实施默认拒绝的授权策略：除非明确验证了所有权或基于角色的权限，否则拒绝所有请求。
  - 为所有资料访问尝试添加审计日志记录，标记已认证用户与请求uid不匹配的情况。
- 工程加固:
  - 实施集中式授权中间件，为每个敏感端点验证资源所有权。
  - 采用基于角色的访问控制（RBAC）或基于属性的访问控制（ABAC），并明确定义权限。
  - 使用间接引用映射：将会话/用户令牌映射到内部ID，而不是暴露直接的对象标识符。
  - 设计API使用路径参数（如 /api/v1/user/profile，从认证令牌自动派生），而非使用带有可变ID的查询参数。
  - 创建安全单元测试，验证当用户尝试访问其不拥有的资源时授权失败。
- 监控与检测:
  - 实施针对IDOR模式的实时监控：对顺序ID访问模式或快速的uid参数变化发出警报。
  - 记录所有敏感数据访问，包含用户上下文、请求的资源ID和授权决策，用于审计追踪。
  - 部署异常检测以识别跨账户访问尝试（例如，用户访问多个不相关的资料）。
  - 定期进行自动化安全扫描，使用已认证和未认证的探针测试访问控制缺陷。
  - 建立安全回归测试，确保已修复的漏洞在代码变更后保持修补状态。
- 建议来源: llm_mitigation_pass
